# 🤝 Contributing Guide (Hackathon Edition)

**ByO Target Platform (TVA)**에 기여해주셔서 감사합니다!  
이 문서는 **해커톤 환경**에서 빠르고 안전한 협업을 위한  
**작업 흐름, 협업 규칙, 커밋 컨벤션**을 정리한 가이드입니다.

> ❗ **본 프로젝트는 기본적으로 Pull Request(PR) 기반으로 작업합니다.**  
> 해커톤 특성상 *속도와 안정성*을 최우선으로 합니다.

---

## 🌿 브랜치 전략

### 🔒 **main 브랜치 규칙**
* `main` 브랜치 **직접 push 지양** (원칙적으로 PR 사용)
* 모든 변경 사항은 **PR을 통해 병합**하는 것을 권장
* 해커톤 상황상 **최소 1명 리뷰를 권장**하되,  
  긴급한 경우 간단한 self-check 후 병합 가능

> 🎯 목적: **데모 안정성 유지 + 사고 방지**

---

### 🌱 **브랜치 네이밍 규칙**
형식: `type/작업-내용`

| 타입 | 설명 | 예시 |
| :--- | :--- | :--- |
| **feature** | 새로운 기능 개발 | `feature/login-api` |
| **fix** | 버그 수정 | `fix/token-expire` |
| **docs** | 문서 작업 | `docs/readme-update` |
| **refactor** | 코드 리팩토링 | `refactor/cleanup` |
| **chore** | 설정, CI, 인프라 | `chore/github-actions` |

> 해커톤 중에는 **명확한 브랜치 이름**이 가장 중요합니다.

---

## 🔄 작업 흐름 (Workflow)

1. **main 브랜치 최신화**: `git pull origin main`
2. **작업 브랜치 생성**: `git checkout -b type/작업-명칭`
3. **기능 개발 및 수정**: 로컬에서 작업 진행
4. **커밋**: 컨벤션에 맞춰 커밋 메시지 작성
5. **Pull Request 생성**: 작업 완료 후 GitHub에서 PR 생성
6. **코드 리뷰**: 동료의 리뷰 및 피드백 반영
7. **병합 (데모 안정성 기준으로 판단)**: 승인 완료 시 `main` 브랜치로 병합

---

## 📝 커밋 컨벤션

### 📌 **커밋 메시지 형식**
`type: 내용` (영문/국문 혼용 가능하나 명확하게 작성)

* ✨ **feat**: 새로운 기능 추가
* 🐛 **fix**: 버그 수정
* 📝 **docs**: 문서 수정
* 🎨 **style**: 코드 포맷팅 (코드 로직 수정 없음)
* ♻️ **refactor**: 리팩토링
* 🔧 **chore**: 빌드 설정, 패키지 관리

> **💡 예시**: `feat: 사용자 로그인 API 구현`, `fix: 토큰 만료 처리 로직 수정`

---

## 🔍 Pull Request 및 리뷰 규칙

### ✅ **PR 체크리스트**
* [ ] **변경 내용 요약 작성**
* [ ] 로컬에서 **빌드 및 실행 테스트** 완료
* [ ] 불필요한 설정 파일/로그 포함 여부 확인
* [ ] **커밋 컨벤션** 준수 여부

### 👀 **리뷰 규칙**
* 최소 **1명 이상**의 승인이 있어야 병합 가능합니다.
* 리뷰 반영 후 다시 리뷰를 요청하세요.
* **본인 PR을 본인이 승인**하는 것은 금지합니다.

---

## 🚫 금지 사항
* **main 브랜치 직접 push** ❌
* 히스토리를 파괴하는 **force push** ❌
* .env, secret, key **파일 커밋** ❌
* 정해진 **협업 규칙 및 컨벤션 무시** ❌
